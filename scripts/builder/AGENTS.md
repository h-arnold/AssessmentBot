# Builder Agent Instructions (`scripts/builder`)

Applies when editing `scripts/builder/**` and build-pipeline behaviour.

## 1. Language and Runtime

- Use idiomatic ECMAScript 2024 TypeScript.
- Builder is Node-based tooling (not GAS runtime code).
- Keep imports/exports and typing explicit; avoid implicit `any` flows.

## 2. Pipeline Authority

- Deployable GAS output is generated by builder commands, not manual edits in `build/gas`.
- Treat `build/*` as generated artefacts; preflight intentionally recreates it.

Primary commands:

- `npm run builder:build`
- `npm run builder:run`
- `npm run builder:lint`
- `npm run builder:test`
- `npm run builder:ci`

## 3. Contractual Inputs and Outputs

Inputs:

- `src/frontend`
- `src/backend`
- `scripts/builder/vendor/jsondbapp`
- `scripts/builder/builder.config.json`

Key outputs:

- `build/gas/appsscript.json`
- `build/gas/JsonDbApp.inlined.js`
- `build/gas/UI/ReactApp.html`

## 4. Stage and Error Model

- Preserve the stage-based pipeline contract in `BuildStageId`.
- Failures should surface as `BuildStageError` with accurate stage context.
- Maintain actionable failure messages and deterministic diagnostics.

## 5. Determinism and Safety Requirements

- Preserve deterministic ordering and output stability.
- Keep checksum-relevant behaviour predictable.
- Maintain path safety checks (paths must resolve inside repo root).
- Preserve manifest merge semantics:
  - backend manifest is the base
  - scope/service merge is de-duplicated deterministically

## 6. HtmlService and Global-Symbol Constraints

- Frontend transform must inline assets into HtmlService-compatible output.
- Output validation must continue to reject unresolved asset references.
- Preserve duplicate protected global checks (`Validate`, `JsonDbAppNS`).

## 7. Config and Test Rules

- Before changing TS/ESLint config, read `docs/developer/builder/TypeScriptAndLintConfigHierarchy.md`.
- Delegate test implementation work to `Testing Specialist`.

## 8. Validation and Type Definition Standard

- Use **Zod** as the validation framework for all new and updated builder validation logic.
- Define the Zod schema first, then derive TypeScript types from that schema using `z.infer<typeof ...>` to avoid duplicated type declarations.
- Store validation schemas in a dedicated adjacent schema file (for example `*.zod.ts` or `zodSchemas.ts`) near the code consuming them.
