# Frontend Standards Audit

This inventory fixes the active `src/frontend` surfaces in scope for the loading-state and width-token sweep before implementation sections begin.

Status terms used below:

- **Needs work**: current implementation is not aligned with the standard and should be treated as a later-section target.
- **No change**: no direct standards change is currently required for that surface.
- **Inherits owner**: the surface is in scope, but its relevant behaviour is owned by a parent surface in this sweep.
- **Out of scope**: deliberately deferred for this standards pass.

Branch-status note: the `Needs work` entries below are the historical audit baseline that Sections 3-6 of `ACTION_PLAN.md` used on this branch. Those sections resolved the branch implementation targets, so no additional active frontend surfaces remain non-compliant on this branch and Section 7 is verification/doc-sync only unless a later change touches a new active frontend surface or reopens an audited gap.

## Deliberate deferrals

- Modal width redesign remains out of scope for `BulkCreateModal`, `BulkDeleteModal`, `BulkSetSelectModal`, `BulkSetCourseLengthModal`, `ManageCohortsModal`, `ManageYearGroupsModal`, and the shared inline dialog helpers.
- `AppThemeShell`, `AppShell`, and `AppAuthGate` are retained in the inventory so the scope boundary stays explicit, but they are not direct UI-standard implementation targets for Sections 3-7.
- Backend settings section cards, `ClassesSummaryCard`, and the shared inline dialog helpers are in scope only as sub-surfaces of their owning panels or modals; later sections should update the owner surface first and only change the sub-surface if a concrete standards gap remains after that owner work.
